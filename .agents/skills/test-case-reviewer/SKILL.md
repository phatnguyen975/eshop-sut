# Skill: test-case-reviewer

## 1. Purpose

Second mandatory QA Gate in the workflow. Reviews the complete TC table against `theory-testcase-design.md` standards and flags all issues before execution begins. Do NOT start execution if any Critical violations remain unresolved.

## 2. Input Required

- File `qa-artifacts/test-cases/FR{nn}-test-cases.md` (generated by test-case-generator)
- **Reference:** `.agents/context/theory-testcase-design.md`

## 3. Review Checklist — 3 Tiers

### Tier 1: CRITICAL — Must Fix Before Execution

These violations can cause false results or missed real bugs:

**C1. Isolation Rule Violation**

- Scan every invalid TC
- Flag: TC has more than 1 invalid input simultaneously
- Fix: split into 2 separate TCs

**C2. Vague Expected Result**

- Flag any Expected Result containing: "correctly", "properly", "successfully" without specific measurable details following them
- Flag any Expected Result missing a FR/SEC citation
- Fix: rewrite with HTTP status, response body, and UI behavior

**C3. Missing FR Citation**

- Every Expected Result must include "(per FR-xx)" or "(per SEC-xx)"
- Flag if absent

**C4. Defect Masking Risk**

- Check: in invalid TCs, are all other inputs truly VALID?
- Flag if a supposedly valid input in an invalid TC is actually invalid

**C5. Missing TC for an identified EC**

- Cross-reference the EC list from domain-analysis with the TC list
- Flag any EC that has no corresponding TC

### Tier 2: SERIOUS — Should Fix

**S1. Bad Objective Title**

- Flag objectives not following `Action + Function + Condition` syntax
- Flag objectives under 5 words or overly generic
- Suggest rewrite

**S2. Incomplete Pre-condition**

- Flag pre-conditions missing auth state, data state, or SUT state
- Flag pre-conditions written as steps instead of state descriptions
- Fix: rewrite as a state description

**S3. Non-concrete Test Data**

- Flag test data with placeholders like `[valid email]` or `<password>`
- Fix: replace with concrete values like `"testuser@gmail.com"`

**S4. Missing Teardown**

- Flag any TC that creates persistent data (user, coupon, cart item, order) but has no teardown step
- Fix: add `[Teardown]` step

**S5. Wrong Test Channel**

- Flag TC testing DOM behavior but assigned to API channel
- Flag TC testing role bypass but assigned to UI channel
- Flag TC assigned to DOM channel if it's the mobile's feature (DOM is N/A for Mobile)
- Fix: correct the channel assignment

**S6. Missing BVA TC**

- Cross-reference BVA points from boundary-analysis with BVA TCs
- Flag any BVA point without a corresponding TC

### Tier 3: COSMETIC — Nice to Fix

**N1. Inconsistent TC ID format**

- Verify all IDs follow `FR{nn}-EP-{nnn}` or `FR{nn}-BVA-{nnn}` exactly

**N2. Steps not numbered**

- Steps must be a numbered list, not a paragraph

**N3. Inconsistent language**

- All content must be in English per project standards

## 4. Step-by-Step Instructions

### Step A — Scan Each TC Against the Checklist

For each TC, complete this quick review:

```
TC ID: FR{nn}-EP-xxx
  [C1] Isolation Rule:         PASS / FAIL (describe violation)
  [C2] Expected Result clear:  PASS / FAIL (describe issue)
  [C3] FR citation present:    PASS / FAIL
  [S1] Objective syntax:       PASS / WARNING (suggest rewrite)
  [S2] Pre-condition complete: PASS / WARNING
  [S3] Concrete test data:     PASS / WARNING
  [S4] Teardown present:       N/A / PASS / MISSING
  [S5] Correct channel:        PASS / WARNING
```

### Step B — Build Coverage Matrix

| EC / BVA Point  | Description            | Covering TC    | Status  |
| --------------- | ---------------------- | -------------- | ------- |
| EC01            | Valid email, not in DB | FR{nn}-EP-001  | Covered |
| EC02            | Invalid email format   | FR{nn}-EP-002  | Covered |
| BVA password LB | 8 chars exact          | FR{nn}-BVA-003 | Covered |

### Step C — Compile Findings by Tier

Group all findings into Critical, Serious, and Cosmetic sections.

### Step D — Calculate Quality Score

```
Total TCs reviewed:    {n}
Critical violations:   {n}
Serious warnings:      {n}
Cosmetic issues:       {n}
Verdict: APPROVED (0 critical) / NEEDS REVISION ({n} critical violations)
```

- [ ] All generated content is completely in English
- [ ] Every TC scanned — none skipped
- [ ] EC list from domain-analysis cross-referenced with TCs
- [ ] BVA points from boundary-analysis cross-referenced with TCs
- [ ] All Critical violations flagged with clear "Required Fix" instructions
- [ ] Coverage matrix is complete
- [ ] Verdict is unambiguous: APPROVED or NEEDS REVISION
- [ ] A preview has been shown to the human, and explicit `APPROVE` command was received before saving to disk
